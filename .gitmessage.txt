# --- BRANCH NAMING GUIDE ---
# 브랜치명은 다음 형식을 사용합니다.
#   <type>/<issue-number>/<description>
#
# type:
#   feature  : 기능 개발
#   fix      : 버그 수정
#   refactor : 리팩토링
#   docs     : 문서 수정
#   test     : 테스트 코드
#   chore    : 설정, 빌드, 의존성 등 기타 작업
#
# examples:
#   feature/12/user-entity
#   feature/13/login-api
#   fix/21/login-error
#   refactor/32/user-service
#   docs/40/update-readme
#   chore/50/init-project-setting
#
# description:
#   - 영어 소문자 권장
#   - 단어 사이는 하이픈(-) 사용
#   - 한글 브랜치명은 가능하면 피합니다.
#
# --- COMMIT MESSAGE GUIDE ---
# 커밋 메시지는 다음 형식을 사용합니다.
#   <type>: <subject>
#
# type:
#   feat     : 새로운 기능
#   fix      : 버그 수정
#   refactor : 리팩토링
#   style    : 코드 포맷팅, 세미콜론, 공백 등
#   docs     : 문서 수정
#   test     : 테스트 코드 추가/수정
#   chore    : 빌드, 설정, 패키지 매니저 등 기타 작업
#
# subject:
#   - 50자 이내 권장
#   - 마침표 금지
#   - 명령문 형태 권장
#
# body:
#   - 무엇을, 왜 변경했는지 작성
#   - 여러 줄 작성 시 '-' 사용 가능
#
# footer:
#   - 관련 이슈가 있다면 작성
#   - ex) Resolves: #12
