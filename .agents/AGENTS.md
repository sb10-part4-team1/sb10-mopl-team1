# Rules

## 1. 담당 개발 범위
- 사용자가 별도의 언급을 하지 않는다면, 담당 개발 범위는 **콘텐츠 도메인(`com.sb10.mopl.content`)** 및 **외부 API를 이용한 콘텐츠 수집 도메인(`com.sb10.mopl.batch`)**입니다. 모든 관련 논의, 코드 수정, 분석 작업 시 이 두 패키지를 기본 타겟으로 설정합니다.

## 2. 코드 스타일 및 빌드 검증 규칙
- 이 프로젝트는 코드 스타일 검증 도구로 **Spotless** 및 **Checkstyle**을 사용합니다.
- 코드를 수정하였을 경우, 반드시 아래의 명령어를 사용하여 빌드와 함께 스타일 검증을 성공적으로 마쳐야 합니다.
  ```bash
  ./gradlew clean build
  ```

## 3. 테스트 코드 작성 스타일
- 모든 테스트 코드는 **BDD(Behavior-Driven Development)** 스타일로 작성되어야 합니다.
- **주석 활용**: 테스트 내부에 `// given: ...`, `// when: ...`, `// then: ...` 주석을 주석 내용에 대한 설명과 함께 필수로 명시하여 논리적 흐름을 구분해야 합니다.
- **메서드 명명 규칙**: 테스트 메서드의 이름은 `[대상메서드]_[기대행동]_[상황/언제]` 구조로 명명합니다.
  - 예시: `createContent_success_whenDataIsValidAndTagsAreNew()`
